**Target:** Risk Engine Developer / Data Scientist  
**Environment:** `https://midas-trade.mooo.com`

## 1. Overview
Торговый бот Midas теперь поддерживает асинхронный процесс валидации сделок, называемый **Risk Engine Approval Flow**. 
Вместо моментального открытия сделки по рынку, бот сначала пересылает распарсенные данные сигнала в специальный канал и переходит в статус `pending`. От Risk Engine требуется произвести анализ сигнала (например, через механизмы секвентинга Monte Carlo) и вернуть решение обратно боту через webhook.

---

## 2. Получение сигнала на проверку (Request)
Бот отправляет все сырые данные по сделке в Telegram-чат **`@uebot_report`**.

Ваш сервер (например, через `estafeta_bot`) должен прослушивать этот чат на предмет новых сообщений от торгового бота. Сообщение имеет строгий текстовый формат.

### Пример сообщения:
```text
🔔 Request for approval
hash: abc123def456

Symbol: ETHUSDT
Side: Buy (LONG)
Entry: 1988.34
TP1: 1995.00 | TP2: 2010.00 | TP3: 2025.00
SL: 1975.00

--- Raw Signal ---
ETHUSDT.P 60M - 🟢 Strong BUY ▲
Win Rate: 85% | R:R: 1:3 ...
[весь оригинальный текст сигнала от Midas]
```
> **Внимание:** Ключевым полем здесь является `hash`. По этому уникальному идентификатору торговый бот находит у себя отложенную сделку. Секция `--- Raw Signal ---` добавлена для того, чтобы Risk Engine мог извлекать дополнительные метаданные (Win-Rate, R:R и т.д.) без лишних запросов.

## 3. Отправка решения (Callback Webhook)
После завершения расчетов, Risk Engine должен отправить HTTP POST запрос на webhook торгового бота с принятым решением.

### Endpoint
```http
POST https://midas-trade.mooo.com/api/re/callback
```
**Headers:**
```http
Content-Type: application/json
```

### JSON Body (Реквизиты запроса)
Структура данных для отправки:

```json
{
    "hash": "abc123def456",
    "decision": "approve", 
    "score": 0.85,
    "reason": "VaR within limits, good trend",
    "secret": "123QWEasd"
}
```

| Поле | Тип | Обязательность | Описание |
| :--- | :--- | :--- | :--- |
| `hash` | `string` | **Required** | Уникальный хэш сигнала, полученный из текста сообщения. |
| `decision` | `string` | **Required** | Решение RE. Допустимые значения: `"approve"` (одобрено) или `"reject"` (отклонено). |
| `score` | `float` | Optional | **Динамический сайзинг:** Если передано, бот использует это значение как множитель для объема позиции. Например, `0.5` откроет сделку на 50% от стандартного риска. |
| `reason` | `string` | Optional | Краткий комментарий/причина. Бот перешлет это администратору в лог. |
| `secret` | `string` | **Required** | Секретный токен для авторизации запроса. Текущее значение: `123QWEasd`. *(Токен задается в `.env` бота)* |

## 4. Обработка ответов (Responses)
После отправки POST запроса, торговый бот вернет стандартный HTTP ответ. Рекомендуется логировать ошибки на стороне Risk Engine.

* **`200 OK`**  
  `{"status": "approved", "symbol": "ETHUSDT"}` — Запрос успешен. Сигнал одобрен, бот **открывает сделку** на Bybit.
* **`200 OK`**  
  `{"status": "rejected", "symbol": "ETHUSDT"}` — Запрос успешен. Сигнал отклонен, бот игнорирует сделку.
* **`403 Forbidden`**  
  `{"error": "invalid secret"}` — Ошибка авторизации (неверное поле `secret`).
* **`404 Not Found`**  
  `{"error": "signal not found or already processed"}` — Сигнал с данным `hash` не найден в статусе `pending` (возможен таймаут, см. ниже).

---

## 5. Таймауты и Жизненный цикл (Timeout Lifecycle)
На стороне торгового бота действует жесткий процесс сборки мусора для неотвеченных сигналов.

* **Таймаут:** `300 секунд` (5 минут).
* Если Risk Engine не успевает прислать ответ (или webhook не дошел) в течение **5 минут** с момента отправки оригинального сообщения в группу, торговый бот автоматически помечает сделку как **`timeout`** и сделка **сбрасывается** (авто-реджект).
* Вызовы webhook для сигналов, ушедших в `timeout`, будут возвращать `404 Not Found`.

---

## 6. Ответственность сторон
1. **Risk Engine** принимает финальное решение о входе в сделку (`approve` / `reject`).
2. **Trading Bot** полностью заведует исполнением (Execution). 
   * Он самостоятельно выставляет *маркет-ордер* на вход.
   * Он самостоятельно расставляет *нативный Stop-Loss* на бирже.
   * Он самостоятельно управляет тремя частями *Take-Profit* (TP1, TP2, TP3) через собственный WebSocket-мониторинг цен Bybit. 
   **Risk Engine не нужно переставлять стопы или рассчитывать цены выхода.**
