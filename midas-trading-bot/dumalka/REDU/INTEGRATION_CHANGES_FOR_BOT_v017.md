# Изменения интеграции Risk Engine v0.17.0 — Для разработчика бота (Дмитрий)

> **Дата:** 1 апреля 2026
> **Автор:** Risk Engine Team
> **Версия RE:** v0.17.0 "Critical Trade Fix"

---

## Обзор

Risk Engine v0.17.0 содержит 12 критических исправлений, направленных на устранение
избыточных стоп-лоссов и утечки прибыли. **Большинство изменений внутренние и не
требуют модификации бота.** Ниже описаны изменения, которые затрагивают интеграцию.

---

## 1. Keepalive команды (НОВОЕ)

### Что изменилось

RE теперь отправляет `move_sl` с текущим значением SL **каждые 5 минут** для каждой
активной позиции, даже если никакие пороги не сработали. Это предотвращает fallback
бота на собственную логику TP2→breakeven.

### Что нужно от бота

**Ничего.** Бот уже корректно обрабатывает `move_sl` с тем же SL — это no-op на стороне
Bybit. Просто убедитесь, что `_should_use_dumalka()` обновляет `last_command_time` при
получении любой команды, включая `move_sl` с неизменённым SL.

### Формат запроса (без изменений)

```json
{
  "action": "move_sl",
  "symbol": "ETHUSDT",
  "new_sl": 2400.0,
  "reason": "keepalive",
  "trace_id": "abc123"
}
```

Поле `reason: "keepalive"` — информационное. Бот может логировать его отдельно для аналитики.

## 2. Увеличена частота команд управления

### Что изменилось

- **Grace period** (период, когда RE не трогает позицию) сократился с **2 часов до 30 минут**
- **Timeout** увеличен с **12 часов до 24 часов**
- **Apollo bail** (закрытие застрявших позиций) теперь может **повторяться** каждые 10 минут
  (раньше была одна попытка на позицию)
- **E_pnl exit** порог снижен с 1.5% до 0.5% — больше full_close команд при небольшой прибыли

### Что нужно от бота

**Ничего.** Все команды используют тот же API (`POST /dumalka/command`). Бот может заметить:
- Больше `move_sl` команд в первые 30 мин — 2 часа жизни позиции
- Больше `full_close` команд для позиций с небольшой прибылью (0.5%+)
- Меньше `full_close` по timeout для позиций старше 12ч (теперь таймаут 24ч)

## 3. Рекомендация: мгновенный callback при закрытии позиции (ЗАПРОС)

### Проблема

Когда Bybit закрывает позицию через нативный SL/TP, RE узнаёт об этом только на
следующем цикле поллинга (30 сек) через `GET /dumalka/positions`. За это время RE
может отправить команды для уже закрытой позиции → ошибки "no active trade".

### Запрос к боту

Пожалуйста, убедитесь, что `POST /trade-outcome` отправляется на RE **сразу**
после обнаружения закрытия позиции (`pos_size = 0`), **ДО** ответа на следующий
поллинг `/dumalka/positions`.

```json
POST http://{RE_URL}/trade-outcome
X-Webhook-Secret: {secret}

{
  "hash": "signal_hash_123",
  "event": "sl_hit",
  "symbol": "ETHUSDT",
  "side": "long",
  "price": 2390.5,
  "pnl_pct": -2.5,
  "size_remaining": 0
}
```

**Приоритет:** Высокий. Это устранит ~40% ошибок "no active trade" в аудит-логе.

## 4. Рекомендация: SL→BE триггер на стороне бота (ЗАПРОС)

### Проблема

370 из 371 позиций закрываются в Zone 0, т.е. до того как RE может передвинуть
SL в безубыток. Бот мониторит позиции каждые 3 секунды, RE — каждые 30 секунд.
Нативный SL на Bybit часто срабатывает раньше, чем RE успевает среагировать.

### Запрос к боту (опционально, но очень ценно)

Добавить параметр `auto_be_at_tp1` в логику открытия сделки:

Когда `DUMALKA_MODE=active`, бот при открытии позиции **дополнительно** устанавливает
условный ордер: если цена достигает TP1, автоматически переносить SL на entry + ATR offset.

Это можно реализовать через Bybit Conditional Order API или в `monitor_trade()`:

```python
# В monitor_trade(), после обнаружения TP1:
if dumalka_active and not tp1_be_done:
    atr_offset = 0.002  # или получить от RE
    be_price = entry * (1 + atr_offset) if side == "long" else entry * (1 - atr_offset)
    set_trading_stop(symbol=symbol, stopLoss=be_price)
    tp1_be_done = True
```

**Приоритет:** Критический. Это единственный способ защитить прибыль быстрее,
чем нативный SL на Bybit успевает сработать.

**Альтернатива (если сложно):** RE может передавать рекомендуемый `be_price` при
callback через `/api/re/callback`:

```json
{
  "hash": "abc123",
  "decision": "approve",
  "score": 0.78,
  "reason": "...",
  "secret": "...",
  "auto_be_price": 2455.0,
  "auto_be_trigger": 2520.0
}
```

Бот должен запомнить `auto_be_price` и `auto_be_trigger` и при достижении
`auto_be_trigger` выполнить `set_trading_stop(stopLoss=auto_be_price)`.

## 5. Никаких partial_close — только full_close

### Что изменилось

RE v0.17.0 **не отправляет** `partial_close` команды. Все закрытия через `full_close`.

### Что нужно от бота

**Ничего.** Endpoint `partial_close` может остаться, но он не будет вызываться.
