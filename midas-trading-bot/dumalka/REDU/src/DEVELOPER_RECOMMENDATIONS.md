## 1. Введение: Суть Active Mode
Мы ушли от старой схемы (Shadow Mode v2), где бот просто спрашивал у Risk Engine "какой скор?" перед входом в сделку. 
Текущая архитектура **v0.14.2 Active Mode** (кодовое имя: Думалка) подразумевает, что Risk Engine берет на себя **Активное Управление Позицией** (HFT). Как только сигнал одобрен (`approve`) и позиция открыта, Risk Engine (GPU Titan V) начинает поллить Bybit каждую микросекунду и генерировать команды на экстренную фиксацию прибыли, если рынок разворачивается.

Твоя задача: **Научить Midas-бота 100% послушно принимать и исполнять команды от Risk Engine**.

## 2. API Контракт: Команды от Risk Engine

Risk Engine шлет команды на эндпоинт бота:
**`POST /dumalka/command`**

### Ожидаемый JSON Payload
Каждая команда содержит поле `action`. Бот **ОБЯЗАН** поддерживать 5 типов `action` без исключений.

#### 1. Установка Лимитной Сетки (`place_limit_tp`)
*Новое в v0.14.2!*
Сразу после открытия позиции RE пытается выставить умные Maker-лимитки (чтобы сэкономить комиссии Taker-ордеров).
```json
{
  "action": "place_limit_tp",
  "symbol": "RIVERUSDT",
  "fraction": 0.3,
  "target_price": 24.50,
  "reason": "Maker Limit Grid 3R"
}
```
**Действие бота:** Отправить на Bybit команду "Лимитный Reduce-Only ордер по цене target_price на объем = (общий_объем * fraction)".

#### 2. Частичное Закрытие "По Рынку" (`partial_close`)
Срабатывает при резких дампах (Drawdown Tracker спасает прибыль).
```json
{
  "action": "partial_close",
  "symbol": "FARTCOINUSDT",
  "fraction": 0.25,
  "reason": "Zone 2 DD Protection"
}
```
**Действие бота:** **ОБЯЗАТЕЛЬНО привести `fraction` к типу `float()`**. Высчитать `quantity = current_size * fraction` и кинуть рыночный (Market) ордер на закрытие.

#### 3. Адаптивный Безубыток (`move_sl`)
Срабатывает при мягких откатах или после частичного тейка.
```json
{
  "action": "move_sl",
  "symbol": "SIRENUSDT",
  "new_sl": 1.05
}
```
**Действие бота:** Изменить Trading Stop-Loss на бирже Bybit на указанный `new_sl`. Без жестких проверок (RE сам всё вычислил).

#### 4. Обновление Тейка (`move_tp`)
```json
{
  "action": "move_tp",
  "symbol": "TAOUSDT",
  "new_tp": 650.0
}
```
**Действие бота:** Передвинуть TP-ордер.

#### 5. Экстренное Закрытие (`full_close`)
Срабатывает при Circuit Breaker или Time-Decay истечении времени (сделка застряла на >4 часов).
```json
{
  "action": "full_close",
  "symbol": "GRASSUSDT",
  "reason": "Time-Decay Exclusion"
}
```
**Действие бота:** Закрыть позицию 100% по рынку.

## 3. Синхронизация Состояний (Думалка ↔ Бот)

Чтобы RE не спамил слепые команды, он делает проверку реально существующих позиций.
**Твой бот должен предоставлять эндпоинт:**
`GET /dumalka/positions`
* Авторизация: Header `X-Dumalka-Token: <SECRET>`
* Ответ: Список актуальных тикеров (символов), которые прямо сейчас открыты на счете на Bybit и трекаются ботом.

Если бот по какой-то причине отверг сделку при старте (нехватка маржи, KillSwitch), но RE об этом не знает, RE спросит `/dumalka/positions`, не найдет там монету и переведет ее в **"Observe Only" (Бумажный трейдинг)**, прекратив спамить командами на `/dumalka/command`.

## 5. Fallback Protection (Защита от отключения)
Если Risk Engine (VM 107) внезапно падает, или связь через Tailscale обрывается:
1. Сделки **НЕ ДОЛЖНЫ** висеть вечно.
2. В твоем коде (в боте) должен быть fallback-счетчик. Если по открытой сделке в течение **15 минут** не пришло ни одной команды от Думалки (при условии, что `DUMALKA_MODE = active`), бот должен вернуться к классическому, тупому Midas-сценарию (статичные TP1/TP2/TP3 и жесткий SL).
