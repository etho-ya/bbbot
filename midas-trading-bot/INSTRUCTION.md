# Midas Trading Bot — инструкция по работе

Подробное описание архитектуры, технологий, базы данных и сценариев работы бота.

---

## 1. Назначение бота

Бот подключается к Telegram под **пользовательским аккаунтом** (не бот-токен), получает торговые сигналы от **внешнего бота** (например, бот с сигналами в личку) и по ним выставляет ордера на бирже **Bybit**. Перед открытием сделки сигнал может проверяться **LLM** (OpenRouter). Уведомления о сделках и решениях LLM отправляются в указанный чат/группу тем же Telegram-аккаунтом.

---

## 2. Технологический стек

| Компонент | Технология | Назначение |
|-----------|------------|------------|
| Backend / API | **FastAPI** | HTTP API, WebSocket, страницы настроек и дашборда |
| Telegram (клиент) | **Telethon** | Вход под пользователем (user session), чтение сообщений и отправка уведомлений |
| Биржа | **Bybit** (API v5), клиент **pybit** | Открытие/закрытие позиций, установка SL/TP, баланс |
| БД | **PostgreSQL** + **SQLAlchemy** (async, asyncpg) | Пользователи, настройки, сделки, сигналы, каналы |
| LLM | **OpenRouter API** | Проверка торговых сигналов и вспомогательных сообщений |
| Запуск | **Docker / Docker Compose** | Контейнеры: приложение + PostgreSQL |
| Конфигурация | **.env**, **Pydantic Settings** | API-ключи, ID бота сигналов, чат уведомлений и т.д. |

## 4. База данных (схема)

Бот хранит данные в PostgreSQL. Ниже — основные таблицы и поля.

### 4.1 Таблица `users`

- **id** — первичный ключ.
- **username**, **password_hash** — для входа (в текущей версии один пользователь admin).
- **bybit_api_key**, **bybit_api_secret** — зашифрованные ключи Bybit.
- **is_testnet** — флаг тестовой сети Bybit.

### 4.2 Таблица `settings`

Настройки на пользователя (user_id):

- Торговля: **deposit_percent**, **leverage**, **max_price_deviation**, **default_stop_loss_percent**, **max_open_positions**, **max_daily_loss_percent**.
- Источник сигналов: **signal_bot_id**, **signal_bot_username** (какой бот в Telegram присылает сигналы).
- LLM: **openrouter_api_key**, **openrouter_model**, **llm_validation_enabled**.
- Уведомления: **telegram_notify_chat_id** — куда слать уведомления (числовой ID или юзернейм, например `uebot_report`).

### 4.3 Таблица `signals`

**Сигналы сохраняются в БД.**

Каждое обработанное сообщение от бота-источника записывается в таблицу `signals`:

- **id** — первичный ключ.
- **user_id** — кому предназначен сигнал.
- **raw_text** — исходный текст сообщения.
- **parsed_data** — JSON с распарсенными полями (символ, сторона, вход, цели, стоп, трейлинг и т.д.).
- **channel_id**, **channel_title** — откуда пришло (например, ID бота или канала).
- **signal_hash** — хэш текста для дедупликации.
- **signal_type** — `"trade"` (торговый сигнал) или `"auxiliary"` (денежный поток, импульс и т.п.).
- **is_processed** — был ли по сигналу открыт ордер.
- **llm_decision** — решение LLM: `"approve"` / `"reject"`.
- **llm_reason**, **llm_model** — причина и модель.
- **created_at** — время получения.

В интерфейсе последние сигналы видны на странице **Сигналы** (`/signals`).

### 4.4 Таблица `trades`

- **id** — UUID сделки.
- **user_id**, **signal_id** — связь с пользователем и сигналом из `signals`.
- **symbol**, **side**, **entry_price**, **size**, **leverage** — параметры позиции.
- **tp1**, **tp2**, **tp3**, **sl**, **trailing_stop_pct** — тейки и стоп из сигнала.
- **stage** — этап (OPEN, TP2_REACHED и т.д.).
- **status** — `"active"` / `"closed"`.
- **profit_loss** — P/L при закрытии.
- **closed_at**, **created_at**, **updated_at**.

### 4.5 Таблица `channels`

Используется для подписки на каналы/чаты (если помимо ЛС с ботом подключаются каналы). Поля: **channel_id**, **title**, **signal_keywords** и т.д.

## 5. Сценарии работы

### 5.1 Авторизация Telegram

- Приложение подключается к Telegram как **пользователь** (не бот) по **API ID** и **API Hash** с my.telegram.org.
- Сессия сохраняется в файл (например `mityan_session.session`). Имя задаётся в **TELEGRAM_SESSION_NAME** в `.env`.
- Если сессии нет или она не авторизована, в UI на странице **Настройки** есть блок «Авторизация Telegram»: ввод номера телефона, запрос кода, ввод кода и при необходимости 2FA-пароля. После успешного входа тот же аккаунт используется для приёма сигналов и отправки уведомлений.

### 5.2 Приём сигналов

- Telethon слушает входящие сообщения (личка и при необходимости каналы из `channels`).
- Берутся только сообщения от бота, указанного в **SIGNAL_BOT_ID** / **SIGNAL_BOT_USERNAME** (в настройках или `.env`).
- Текст сообщения классифицируется: **trade** (торговый сигнал с входом, целями, стопом) или **auxiliary** (денежный поток, импульс, киты и т.п.).
- Для каждого нового сигнала создаётся запись в таблице **signals** (raw_text, parsed_data, signal_type и т.д.).

### 5.3 Обработка торгового сигнала (trade)

1. Парсинг текста (символ, направление, вход, цели TP1–TP3, стоп, трейлинг).
2. При включённой LLM — запрос к OpenRouter: проверить сигнал и при необходимости скорректировать парсинг. Результат (approve/reject, reason) пишется в **signals** (llm_decision, llm_reason, llm_model).
3. При **reject** — уведомление в Telegram (причина отклонения), сделка не открывается.
4. При **approve** — вызов Trade Manager: проверки (лимит позиций, дневной убыток), расчёт размера, **только рыночный ордер** на Bybit, установка **SL и TP3** на позиции. Запись в **trades**, обновление **signals** (is_processed).
5. Монитор по позиции: пока цена не достигла TP2 — ничего не меняет; при достижении TP2 — перенос **SL в безубыток** (на цену входа). Закрытие по TP3 или SL выполняет сама биржа.

### 5.4 Вспомогательные сигналы (auxiliary)

- Сообщения типа «Денежный поток растёт/падает», «Импульс разворота», «Киты закончили распределение» и т.п. парсятся как **auxiliary**.
- LLM получает список открытых позиций и текст сообщения и возвращает действие: **do_nothing**, **close**, **tighten_sl** и причину.
- Бот выполняет действие (закрытие позиции или подтяжка SL) и отправляет уведомление.

### 5.5 Уведомления в Telegram

- Все уведомления отправляются **тем же пользовательским аккаунтом** (Telethon), не бот-токеном.
- Куда слать задаётся в настройках: **Chat ID или @username** (например `uebot_report` для публичной группы).
- Поддерживаются:
  - числовой ID чата/группы (например `-100123456789`);
  - юзернейм публичной группы/канала без `@` или с `@` (например `uebot_report` или `@uebot_report`).
- События: сделка открыта/закрыта, достигнут TP2 (SL в безубыток), сработал SL, TP3 достигнут, LLM отклонила сигнал, kill switch, ошибки.

## 6. Конфигурация (.env)

Основные переменные:

- **TELEGRAM_API_ID**, **TELEGRAM_API_HASH**, **TELEGRAM_SESSION_NAME** — вход в Telegram как пользователь.
- **SIGNAL_BOT_ID**, **SIGNAL_BOT_USERNAME** — бот, от которого принимаются сигналы.
- **TELEGRAM_NOTIFY_CHAT_ID** — куда слать уведомления: число или юзернейм (например `uebot_report`).
- **BYBIT_API_KEY**, **BYBIT_API_SECRET**, **BYBIT_TESTNET** — доступ к Bybit.
- **OPENROUTER_API_KEY**, **OPENROUTER_MODEL** — LLM (по умолчанию можно оставить модель из настроек в UI).
- **DATABASE_URL** — строка подключения к PostgreSQL (в Docker подставляется из docker-compose).
- **DEPOSIT_PERCENT**, **DEFAULT_LEVERAGE**, **MAX_OPEN_POSITIONS**, **MAX_DAILY_LOSS_PERCENT** и др. — можно переопределить дефолты из UI.

## 7. Запуск через Docker

```bash
# Сборка и запуск
docker compose up -d --build

# Логи
docker compose logs -f bot

# Остановка
docker compose down
```

- Приложение слушает порт **8000** внутри контейнера; снаружи он может быть проброшен на другой порт (например 8001 в docker-compose).
- PostgreSQL поднимается отдельным контейнером, данные в volume **pgdata**.
- Сессия Telegram хранится в `session_data` внутри контейнера (volume проекта), чтобы не терять авторизацию при пересборке.

## 8. Ответы на вопросы партнёра

**Храним ли мы сигналы в БД?**  
Да. Все обработанные сигналы сохраняются в таблице **signals**: исходный текст, распарсенные данные, тип (trade/auxiliary), решение LLM, время. Последние сигналы отображаются на странице «Сигналы» в веб-интерфейсе.

**Почему перестали приходить репорты в группу?**  
После того как группа стала публичной (@uebot_report), нужно указывать в настройках уведомлений **юзернейм** (например `uebot_report` или `@uebot_report`) вместо старого числового ID. Бот поддерживает оба варианта; для публичных групп достаточно указать юзернейм и сохранить настройки.
