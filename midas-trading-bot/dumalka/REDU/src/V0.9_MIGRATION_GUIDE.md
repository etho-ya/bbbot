## 5. Known Risks & Mitigations (8 total)

### RISK 1: Circular Dependency (main.py <-> position_tracker.py)
- **Threat:** You CANNOT replace one without the other.
- **Fix:** Extract `send_telegram_alert` into `src/notifications.py` FIRST.

### RISK 2: Silent Data Loss on Schema Migration
- **Threat:** v2 schema uses INTEGER timestamps instead of TEXT. Naive migration corrupts data.
- **Fix:** Use `strftime('%s', snapshot_at)` for conversion. Validate row counts before vs after.

### RISK 3: ML Pipeline Interruption
- **Threat:** Any gap in `position_snapshots` collection creates holes in XGBoost training data.
- **Fix:** Run old and new inserts in parallel for 24h before cutover.

### RISK 4: Telegram Signal Loss
- **Threat:** While `telegram_bridge.py` is down, incoming signals are silently dropped = missed trades.
- **Fix:** Refactor `telegram_bridge.py` LAST. It's a separate process, safe to keep old version.

### RISK 5: SQLite WAL Lock
- **Threat:** SQLite allows only one writer. Migration scripts + bot cannot write simultaneously.
- **Fix:** ALWAYS stop the bot before running migration scripts.

### RISK 6: 7 Satellite Modules Not Covered by Prototypes
- **Threat:** `config.py`, `models.py`, `bybit.py`, `regime_detector.py`, `portfolio_allocator.py`, `watchlist_scanner.py`, `core/monte_carlo.py` have no v2 prototypes.
- **Fix:** Do NOT refactor them in v0.9. They have clean APIs, no circular deps. Leave as-is.

### RISK 7: Frontend HTML Hardcoded to API Routes
- **Threat:** `dashboard.html`, `analytics.html`, `info.html` contain hardcoded `fetch()` URLs. Renaming endpoints breaks frontend SILENTLY.
- **Fix:** Keep ALL API route paths identical in new routers. Verify HTML pages after each phase.

### RISK 8: position_manager.py vs position_tracker.py Overlap — ✅ RESOLVED (v0.13.1)
- **Threat:** Two files with similar names, overlapping responsibility.
- **Fix:** ~~Analyze both before migration. One must absorb the other.~~ **RESOLVED**: `position_manager.py` archived to `src/_archive/` — confirmed 100% subset of `position_tracker.py` (4 imported symbols used nowhere).

> Some deliver massive ROI for low risk. Others are high risk with mainly cosmetic benefit.

### Priority Matrix
| Component | Risk | ROI | Verdict |
|---|---|---|---|

| `db.py` indices | ZERO | **HIGH** (50x ML speedup) | **DONE (applied 2026-03-22)** |

### TIER 1: Minimum Viable Migration (v0.9-lite)
**Risk: LOW. Downtime: ZERO. Time: ~1 hour. Gets 60% of total benefit.**

1. Create `src/notifications.py` - extract `send_telegram_alert()`.
   Update `position_tracker.py` to import from it.
   **Result: Circular dependency BROKEN. Biggest architectural win.**

2. Port `scoring.py` to Strategy pattern (see `scoring_v2_prototype.py`).
   Run old + new scorer on same input, assert identical output.
   **Result: XGBoost can be plugged in. A/B testing ready.**

3. Verify: Bot running? Dashboard works? ML snapshots growing?

> **STOP GATE: Stop here if risk appetite is low. System already improved.**

---

### TIER 2: Data Layer + Telegram (When building ML pipeline)
**Risk: MEDIUM. Downtime: ~5 min. Time: ~2 hours.**

1. Stop bot. Take fresh DB backup.
2. Wrap `db.py` functions in Repository classes (see `db_v2_prototype.py`).
   Keep the **SAME SQL** inside. This is wrapping, not rewriting.
3. Port `telegram_bridge.py` (separate process, safe). Split regex from HTTP.
4. Restart. Verify 24h stability.

> **STOP GATE: Stop here unless main.py is actively blocking development.**

---

### TIER 3: Big Surgery (Only when main.py becomes unmaintainable)
**Risk: HIGH. Downtime: ~15 min. Time: ~3 hours. 80% of regression bugs live here.**

1. Port `position_tracker.py` + `main.py` TOGETHER (coupled, must migrate simultaneously).
2. KEEP ALL URL PATHS IDENTICAL (frontend depends on them!).
3. Run 1h with active monitoring: all 3 HTML pages, snapshots growing, /health OK.

> **Honest warning:** Only do Tier 3 when the 1842-line main.py is actively
> blocking development. The monolith WORKS. Don't fix what isn't broken.
